Mechatronic Design Project 2
Team of 4 | 4 weeks | 2018

Working in a team of 4 a challenge was set to design and build a robot with both remote manual and autonomous control modes. To add complexity to the project the robot would be sealed within a transparent plastic ball, all electronics would be mounted on a bespoke PCB and behaviour would be augmented using machine vision.
Initial robot architecture sketches set up the positioning of main components such as the battery, motors, wheels, PCB and camera. An opportunity was identified to use a direct drive with low centre of mass to improve the performance of the robot.
The robot's design was developed to embodiment through a cycle of prototypes and iterative improvement.

During initial testing a number of batteries died very quickly. To debug the problem, I used an oscilloscope to observe the voltage at the battery terminals when accelerating the robot. A diode was then added to prevent back emf being transmitted to the battery.
Voltage and back emf spike when accelerating before and after adding diode to PCB

After observing the limited absolute positional accuracy that could be achieved using open- or closed-loop motor positional control when driving the robot in its ball, the team decided to implement a "coordinate control" system. In this mode the robot would infer its absolute cartesian coordinates using machine vision to identify arUco codes positioned around an arena. This current position would be compared with the desired position to which the robot would move.
Velocity, Position and Coordinate control modes, respectively

The coordinate control mode would implement closed-loop real-world positional control, automatically correcting errors. The robot was programmed to communicate with a laptop terminal running a python GUI via WiFi, with the deployed control code written entirely in simulink
The required distance change was passed to a positional PI block (1), which was passed along with the required angular change to block (2), which modified the desired velocity for left and right DC motors. A conventional closed-loop velocity control system was then implemented in stage (3). In stage (4), depending on the signage of the PWM signal the PWM pins were switched to achieve directional control.
The python GUI allowed the user to remotely interface with the robot. Although the coordinate mode control paremeters were not fully tuned, the final demonstration displayed the robot's position within the area on the python GUI.
